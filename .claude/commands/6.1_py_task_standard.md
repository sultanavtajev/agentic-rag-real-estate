## Arbeidsmetodikk
- Intervju meg ved å bruke AskUserQuestion for å bygge spec
- La oss gå gjennom funksjonaliteten fra start til slutt som en bruker. Steg én: hva ser vi og hva bør vi se. Steg to: hva ser vi og hva bør vi se. Og slik videre med så mange steg som behøves.
- La oss dele denne oppgaven i så små deler som mulig.
- Vær grundig.
- Atomære enheter: Én komponent/funksjon = én oppgave

## Sub-agents
- Bruk sub-agents for parallelle eller uavhengige oppgaver som kan utføres samtidig
- Deleger til sub-agents når:
  - Oppgaver kan kjøres uavhengig av hverandre
  - Det er research/analyse som ikke blokkerer hovedflyten
  - Testing eller validering av isolerte komponenter
  - Generering av dokumentasjon eller typer
- Hver sub-agent får ett klart, avgrenset ansvar
- Koordiner resultater fra sub-agents før neste hovedsteg

## Kodestruktur
- Les `docs/GUIDELINES.md` for alle retningslinjer (filstørrelser, navngivning, typesikkerhet, feilhåndtering, kodestil)
- Python-moduler: `src/{modul}/` — følg eksisterende struktur i `src/`
- API-endepunkter: `api/routers/` med FastAPI
- Frontend: `frontend/` med Next.js
- Pydantic `BaseModel` for alle data som krysser modulgrenser (ingen rå `dict`)
- Fullstendige type annotations på alle public funksjoner
- Sjekk modulsjekklister i `docs/0.4_prosjekt_oppsett/` for krav til den aktuelle modulen
- Sjekk at alle deler som blir påvirket av endringen blir tilpasset, både backend og frontend

## UI/UX (kun ved frontend-arbeid)
- Alt UI/UX skal være responsivt og fungere på alle skjermstørrelser (mobile-first)
- Alt skal være strukturert, oversiktlig og intuitivt
- Bruk Shadcn-komponenter hvis det finnes ferdige. Lag nye komponenter i Shadcn-stil ved behov.

## Kvalitet
- Python: `ruff check . --fix` + `ruff format .`
- TypeScript: `npm run lint` (kun ved frontend-endringer)
- Kjør `pytest` og verifiser at alle tester passerer
- Bruk ekte API-kall i tester for å verifisere reell oppførsel
- Mål: 80% testdekning på `src/` og `api/`

## Sjekkliste
- Opprett én `.md`-fil som sjekkliste under `docs/teknisk/sjekkliste/` med beskrivende filnavn
- Referer til sjekklisten og kryss av etterhvert som vi implementerer endringer
- Marker oppgaver som egner seg for sub-agents med [SUB-AGENT] i sjekklisten

## Når alt i sjekklisten er ferdig implementert
- Kjør `ruff check . --fix` og `ruff format .`
- Kjør `pytest` og verifiser at alle tester passerer
- Commit alle endringer via `/3.0_commit_all`
- Ikke merge til andre brancher og ikke push
- Legg til "OK - " først i navnet på den gjeldende sjekkliste-filen
